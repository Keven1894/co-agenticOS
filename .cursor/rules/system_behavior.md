# System Behavior Rules

## üéØ Core Identity

You are contributing to **co-agenticOS** - a **Framework Documentation & Rule Development Project**.

### Project Mission
co-agenticOS is a Rule & Template Framework for human-AI collaboration in software engineering. It defines culture, workflows, and shared rituals for co-creation between engineers and AI agents.

### Project Type
- **NOT a codebase** - This is a documentation and rule framework project
- **NOT an application** - No code execution or software deployment
- **IS a meta-framework** - Provides patterns and templates for other projects
- **IS a knowledge repository** - Accumulates AI collaboration best practices

## üîß Engineering Behavior

### Documentation-First Approach
- All content is documentation (Markdown files)
- Quality over quantity - every rule must be well-crafted
- Clarity is paramount - rules must be actionable and understandable
- Consistency matters - follow established patterns and structures

### Before Creating or Modifying Content
- Review existing rules and templates to understand patterns
- Check if similar content already exists
- Ensure your contribution fits the framework structure
- Ask clarifying questions if requirements are unclear

### Quality Standards
- Use clear, professional English
- Follow Markdown formatting standards
- Include proper frontmatter metadata
- Provide concrete examples and anti-patterns
- Add validation checklists where appropriate

## üìù Output Standards

### Documentation Creation
- Use templates from `templates/` directory as starting points
- Follow the established directory structure
- Include complete sections (don't leave placeholders)
- Provide realistic, working examples
- Add proper cross-references to related content

### Content Organization
- **Core Rules** (`core/`) - Universal, domain-independent principles
- **Domain Rules** (`domains/`) - Specific to particular fields
- **Templates** (`templates/`) - Structural scaffolds for creating content
- **Examples** (`examples/` and `domains/*/examples/`) - Real-world implementations

## üåê Language Policy - CRITICAL

### English-Only Documentation
- **ALL documentation, rules, templates, and examples MUST be written in ENGLISH**
- **NO exceptions** - Regardless of discussion language, final output is English
- **NO mixed languages** - Don't mix Chinese, Spanish, or other languages in documentation
- **Comments and explanations** - Also in English
- **Commit messages** - English only

### Why English-Only?
- **Global accessibility** - Framework used worldwide
- **Professional standard** - Industry standard for technical documentation
- **Consistency** - Ensures uniform quality across all content
- **Searchability** - Better discoverability and indexing

### Acceptable Non-English Content
- **Author names** - Can include non-English characters
- **Citations** - Can reference non-English sources (with English translation)
- **Discussion** - Internal discussions can be in any language, but final output is English

## üìã Content Development Discipline

### Follow Established Patterns
- Study reference implementations before creating new content
- Use the GIS Agent Platform example as a quality benchmark
- Maintain consistency with existing rule structures
- Don't reinvent patterns that already exist

### No Autonomous Restructuring
- Do NOT reorganize directories or files without discussion
- Do NOT change established patterns without approval
- Do NOT delete or move content without confirmation
- Suggest improvements, but wait for approval before implementing

### Collaborative Approach
- Your role is to enhance and expand the framework
- Discuss significant changes before implementation
- Seek feedback on new patterns or structures
- Respect the established framework architecture

## üìÅ File Naming Conventions

### Markdown Files
- Use **kebab-case**: `rule-development-workflow.md`
- Be descriptive: `cross-domain-workflow-integration.md`
- Avoid abbreviations unless well-known: `ADR.md` (Architecture Decision Record)

### Directory Names
- Use **kebab-case**: `research-engineering/`
- Be clear and specific: `examples/Agent-dev-GIS-Research/`
- Follow domain naming: `domains/software-development/`

## üéØ Success Criteria

### Quality Indicators
- **Clarity**: Content is easy to understand and actionable
- **Completeness**: All required sections are included
- **Consistency**: Follows established patterns and standards
- **Examples**: Provides realistic, working examples
- **Validation**: Includes checklists or success criteria

### Framework Health
- **Coherence**: New content integrates well with existing framework
- **Usability**: Users can easily find and apply rules
- **Maintainability**: Content is easy to update and improve
- **Growth**: Framework expands without compromising quality
